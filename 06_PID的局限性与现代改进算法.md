# 第六章：PID 的局限性与现代改进算法（前沿研究）

---

## 6.0 引子：当 PID 开始"力不从心"

经过前五章的学习，你已经能熟练地用 PID 让电机稳速、让无人机悬停、让机械臂到位。但当你面对以下场景时，经典 PID 可能会让你陷入"调参地狱"：

- **机械臂高速抓取**：低速时 $K_p=2.0$ 刚好，高速时同样的参数却让末端剧烈抖动——系统动力学随速度变化。
- **化工反应釜升温**：50°C 时加热快，150°C 时散热加剧，升温曲线完全不同——系统特性非线性且时变。
- **六旋翼飞行器做大机动**：roll 和 pitch 的耦合在急转弯时变得不可忽略——多输入多输出（MIMO）的耦合效应。

这些问题的根源不是 PID"坏了"，而是**经典 PID 的假设与现实世界不匹配**。本章将带你走到 PID 的边界之外，看看学术界和工业界如何用"改进型 PID"突破这些限制——同时保持嵌入式可实现的清醒认知。

---

## 6.1 传统 PID 的四道边界

在深入改进算法之前，先明确经典 PID 的"四大约束"：

| 约束 | 含义 | 嵌入式场景中的表现 |
|------|------|-------------------|
| **线性假设** | PID 基于线性系统理论设计 | 电机饱和、摩擦死区、传动间隙导致线性近似失效 |
| **SISO 限制** | 单输入单输出 | 机械臂多关节耦合、无人机姿态-位置耦合 |
| **时不变假设** | 参数固定后不再改变 | 负载突变、电池电压下降、温度漂移 |
| **参数固定** | $K_p, K_i, K_d$ 为常数 | 一套参数无法覆盖全工况（如空载/满载电机） |

> **工程现实**：80% 的嵌入式场景用经典 PID 足够。但当系统出现"某个工况下很好，换个工况就崩"的现象时，你就需要本章的改进方案。

## 6.2 模糊 PID（Fuzzy PID）：把老师傅的经验写进代码

### 6.2.1 核心思想

经典 PID 调参依赖精确的数学模型或大量试错。但很多老师傅凭经验就能调好一个锅炉或机床——他们不看公式，而是观察"误差大不大""变化快不快"，然后凭直觉调整力度。

**模糊 PID 就是把这种人类经验编码成规则。**

它不直接计算 $K_p \cdot e$，而是先问：

- 误差 $e$ 是"大"还是"小"？
- 误差变化率 $\dot{e}$ 是"快"还是"慢"？

然后根据**模糊规则表**查表得到修正后的 $K_p, K_i, K_d$。

### 6.2.2 一个直观的例子

规则表（简化版）：

| 误差 $e$ | 误差变化 $\dot{e}$ | $K_p$ 调整 | $K_i$ 调整 | $K_d$ 调整 |
|---------|-------------------|-----------|-----------|-----------|
| 大 | 快 | 很大（快速逼近） | 零（防饱和） | 中等（抑制超调） |
| 大 | 慢 | 大 | 小 | 小 |
| 小 | 快 | 小 | 中等 | 大（强阻尼防振荡） |
| 小 | 慢 | 很小 | 大（精细消除静差） | 中等 |

> **物理直觉**：离目标还很远且冲得很快（大误差+快变化）→ 大力出奇迹，但要准备刹车。快到目标了且变化慢（小误差+慢变化）→ gently 消除静差。

### 6.2.3 嵌入式实现：查表法

完整的模糊推理（模糊化→规则匹配→解模糊）计算量很大，不适合普通 MCU。工程上的实用做法是**离线计算，在线查表**。

```c
// 定义误差和误差变化率的模糊等级
#define E_NEG_LARGE  0
#define E_NEG_SMALL  1
#define E_ZERO       2
#define E_POS_SMALL  3
#define E_POS_LARGE  4

#define DE_NEG_FAST  0
#define DE_NEG_SLOW  1
#define DE_ZERO      2
#define DE_POS_SLOW  3
#define DE_POS_FAST  4

// 离线生成的 KP 修正表 [5][5]
const float KP_TABLE[5][5] = {
    {2.0f, 1.8f, 1.5f, 1.2f, 1.0f},  // e=NL
    {1.8f, 1.5f, 1.2f, 1.0f, 0.8f},  // e=NS
    {1.5f, 1.2f, 1.0f, 1.2f, 1.5f},  // e=Z
    {0.8f, 1.0f, 1.2f, 1.5f, 1.8f},  // e=PS
    {1.0f, 1.2f, 1.5f, 1.8f, 2.0f}   // e=PL
};

// 在线运行：根据 e 和 de 查表（或插值）
float Fuzzy_PID_Update(PID_Pos_t *pid, float measurement)
{
    float e = pid->setpoint - measurement;
    float de = e - pid->last_error;
    
    // 量化到等级（根据实际系统设定阈值）
    int e_idx = QuantizeError(e);      // 如 e>100→4, e>50→3...
    int de_idx = QuantizeDeltaError(de); // 同理
    
    // 查表修正基准参数
    float kp_adj = KP_TABLE[e_idx][de_idx];
    float ki_adj = KI_TABLE[e_idx][de_idx];
    float kd_adj = KD_TABLE[e_idx][de_idx];
    
    // 临时调整 PID 参数（或直接用修正系数乘到输出上）
    float original_kp = pid->Kp;
    pid->Kp *= kp_adj;
    
    float output = PID_Pos_Update(pid, measurement);
    
    pid->Kp = original_kp; // 恢复基准值
    return output;
}
```

**嵌入式优势**：

- 查表只需数组索引，微秒级完成。
- 不依赖被控对象数学模型。
- 适合**非线性严重但经验丰富**的系统（如老锅炉、某些液压系统）。

**局限性**：

- 规则表设计需要专家经验。
- 规则数量随变量指数增长，高维系统难以覆盖。

## 6.3 自适应 PID：参数会"见风使舵"

如果说模糊 PID 是"凭经验换参数"，自适应 PID 就是"让系统自己找参数"。

### 6.3.1 增益调度（Gain Scheduling）：最实用的自适应

**核心思想**：系统在不同工作点表现不同，那就**分区管理**。

> **例子**：无人机在悬停（低油门）和全速爬升（高油门）时，电机响应完全不同。可以预先调好两组参数：
>
> - 低油门区： $K_p=1.5, K_i=0.3, K_d=0.05$
> - 高油门区： $K_p=1.0, K_i=0.2, K_d=0.08$（高油门时气动阻尼变化，需降低 P 增大 D）

**嵌入式实现**：

```c
typedef struct {
    float throttle_threshold;
    float Kp, Ki, Kd;
} GainSchedule_Entry_t;

const GainSchedule_Entry_t SCHEDULE[] = {
    {0.0f,  1.5f, 0.3f, 0.05f},   // 低油门
    {0.5f,  1.2f, 0.25f, 0.06f},  // 中油门
    {0.8f,  1.0f, 0.2f, 0.08f},   // 高油门
};

void PID_GainSchedule(PID_Inc_t *pid, float current_throttle)
{
    for (int i = 0; i < 3; i++) {
        if (current_throttle < SCHEDULE[i].throttle_threshold) {
            pid->Kp = SCHEDULE[i].Kp;
            pid->Ki = SCHEDULE[i].Ki;
            pid->Kd = SCHEDULE[i].Kd;
            return;
        }
    }
    // 默认最高区
    pid->Kp = SCHEDULE[2].Kp;
    pid->Ki = SCHEDULE[2].Ki;
    pid->Kd = SCHEDULE[2].Kd;
}
```

**平滑切换**：参数突变会导致控制量跳变。应在切换时用一阶滤波过渡：

```c
pid->Kp += 0.1f * (target_Kp - pid->Kp);  // 每周期平滑过渡 10%
```

### 6.3.2 模型参考自适应（MRAC）：让系统"照猫画虎"

**核心思想**：定义一个理想的"参考模型"（如一个响应快、无超调的二阶系统）。实时比较实际系统与参考模型的输出差异，用自适应律在线调整 PID 参数，使实际输出逼近理想模型。

**数学本质**（Lyapunov 稳定性理论）：

$$
\frac{dK_p}{dt} = -\gamma \cdot e_{model} \cdot e_{system}
$$

其中 $e_{model}$ 是参考模型与系统的误差， $\gamma$ 是学习率。

**嵌入式现实**：

- MRAC 需要实时求解微分方程，计算量较大。
- 在 Cortex-M4F 上可行，但 M0/M3 上吃力。
- 更常见的是在上位机（Linux/PC）做自适应，MCU 只执行固定参数。

## 6.4 神经网络 PID 与强化学习 PID：数据驱动的未来

### 6.4.1 神经网络 PID（NN-PID）

**核心思想**：用神经网络（通常是小型的 BP 网络或 RBF 网络）在线学习系统的逆动力学，或直接输出 PID 参数。

**两种架构**：

1. **直接型**：NN 的输出就是 $K_p, K_i, K_d$。
2. **间接型**：NN 作为前馈补偿器，PID 处理残余误差。

**嵌入式现状**：

- 一个 3-5-3 的 BP 网络，前向传播约 50~100 次乘加。
- 有 FPU 的 Cortex-M4F 可以运行，但训练（反向传播）通常离线完成。
- **在线学习**在 MCU 上仍属前沿，需要定点化、网络剪枝等优化。

### 6.4.2 强化学习 PID（RL-PID）

**核心思想**：把 PID 调参看作一个**马尔可夫决策过程**：

- **状态**：当前误差、误差变化率、系统输出。
- **动作**：调整 $K_p, K_i, K_d$（增大/减小/不变）。
- **奖励**：-（超调量 + 调节时间 + 稳态误差）。

智能体通过与仿真环境交互，学习最优调参策略。

**嵌入式路径**：

- **训练在云端/PC**：用 Simulink 或 Python 训练 RL 策略。
- **推理在 MCU**：把训练好的策略网络（或简化为查找表/决策树）部署到嵌入式。
- 目前多见于研究阶段，工业级应用仍在探索。

> **现实判断**：对于资源受限的嵌入式系统，NN/RL-PID 目前仍是"学术炫技"多于"工程实用"。但随着 TinyML（微型机器学习）的发展，未来 3~5 年可能在高端 MCU 上普及。

## 6.5 分数阶 PID（Fractional-order PID）：打破整数阶的枷锁

### 6.5.1 数学突破

传统 PID 的微积分阶次是固定的整数：积分是 $s^{-1}$，微分是 $s^{1}$。

分数阶 PID（ $PI^\lambda D^\mu$）提出：**为什么积分阶次必须是 1？微分阶次必须是 1？**

$$
G_c(s) = K_p + K_i \cdot s^{-\lambda} + K_d \cdot s^{\mu}
$$

其中 $\lambda, \mu$ 可以是任意正实数（如 0.5, 1.2）。

### 6.5.2 为什么多两个自由度很有用？

经典 PID 有 3 个参数（ $K_p, K_i, K_d$），分数阶 PID 有 5 个（加上 $\lambda, \mu$）。

这额外的两个自由度让控制器可以：

- **更精细地调节相位裕度**：在波特图上更灵活地"塑造"系统频率响应。
- **更好地处理记忆效应**：分数阶积分对历史误差的衰减是幂律型的，而非指数型，适合有记忆特性的系统（如粘弹性材料、锂电池）。

### 6.5.3 嵌入式实现的困境

分数阶算子在连续域没有简单的时域表达式，必须近似为**高阶整数阶系统**（如 Oustaloup 近似法）：

$$
s^{0.5} \approx \frac{as^2 + bs + c}{ds^2 + es + f}
$$

这意味着：

- 一个分数阶 PID 在离散实现时，可能需要保存 **10~20 个历史状态**（而非经典 PID 的 2~3 个）。
- 计算量是经典 PID 的 5~10 倍。
- 内存占用显著增加。

**当前状态**：

- 学术研究非常活跃，在温度控制、电机控制中已证明优于经典 PID。
- 嵌入式实现仍限于高端 DSP 或 FPGA，普通 MCU 上较少见。
- 如果你对理论感兴趣，这是很好的研究方向；如果追求工程落地，建议先掌握经典 PID。

## 6.6 结构改进：MIMO 系统的解耦策略

### 6.6.1 多变量耦合的痛苦

经典 PID 假设一个控制器输出只影响一个被控量。但现实中：

- 无人机的 roll 电机加速时，会产生反扭矩影响 yaw。
- 机械臂肩关节运动时，肘关节的负载力矩会变化。
- 化工精馏塔中，温度和压力互相影响。

### 6.6.2 解耦 PID

**核心思想**：在 PID 输出前加一个**解耦矩阵**，把"耦合的系统"变成"独立的子系统"。

$$
\mathbf{u} = \mathbf{D} \cdot \mathbf{u}_{PID}
$$

其中 $\mathbf{D}$ 是解耦矩阵，通常通过对被控对象的传递函数矩阵求逆得到。

**嵌入式简化**：

精确解耦需要知道系统的完整 MIMO 模型，在嵌入式中往往不可行。工程上更常见的做法是：

1. **对角占优假设**：假设耦合较弱，各轴独立用 PID，只在调试时手动补偿（如无人机的混控矩阵）。
2. **前馈解耦**：把已知耦合当作扰动，用前馈补偿。

```c
// 无人机混控：把 roll/pitch/yaw 指令分配到 4 个电机
// 这本身就是一种解耦
void Motor_Mix(float roll, float pitch, float yaw, float throttle)
{
    motor[0] = throttle + roll + pitch + yaw;  // 前左
    motor[1] = throttle - roll + pitch - yaw;  // 前右
    motor[2] = throttle + roll - pitch - yaw;  // 后左
    motor[3] = throttle - roll - pitch + yaw;  // 后右
    // 归一化限幅...
}
```

## 6.7 嵌入式资源受限下的算法选择决策树

面对这么多改进算法，嵌入式工程师常问："我该用哪个？" 以下决策树基于 **MCU 算力** 和 **系统复杂度**：

```
开始
  │
  ▼
系统是否线性且时不变？
  ├─ 是 → 经典 PID（第三章代码直接可用）
  ▼
是否有明确的工况分区（如空载/满载/高速/低速）？
  ├─ 是 → 增益调度 PID（6.3.1节，查表切换，零额外计算量）
  ▼
是否有丰富专家经验但难以建模？
  ├─ 是 → 模糊 PID（6.2节，查表法，微秒级）
  ▼
MCU 是否有 FPU（Cortex-M4F/M7）？
  ├─ 否 → 经典 PID + 前馈补偿（最稳）
  ▼
系统是否强非线性且性能要求高？
  ├─ 是 → 自适应 PID / MRAC（需浮点运算）
  ▼
是否有上位机/云端支持训练？
  ├─ 是 → NN-PID 或 RL-PID（推理部署到 MCU）
  ▼
资源无限且追求极致性能？
  ├─ 是 → 分数阶 PID（研究性质）
  └─ 否 → 回到经典 PID，把结构做好（串级+前馈）
```

| 算法 | 额外计算开销 | 额外内存 | 嵌入式可行性 | 推荐平台 |
|------|------------|---------|-------------|---------|
| 经典 PID | 基准 | 基准 | ⭐⭐⭐⭐⭐ | 所有 MCU |
| 增益调度 | 无（查表） | 极小 | ⭐⭐⭐⭐⭐ | 所有 MCU |
| 模糊 PID（查表） | 极小 | 中（规则表） | ⭐⭐⭐⭐⭐ | 所有 MCU |
| 前馈补偿 | 小（模型计算） | 小 | ⭐⭐⭐⭐⭐ | 所有 MCU |
| MRAC | 大（微分方程） | 中 | ⭐⭐⭐ | M4F/M7 |
| NN-PID（推理） | 中（矩阵运算） | 中 | ⭐⭐⭐ | M4F/M7 |
| 分数阶 PID | 很大（高阶近似） | 大 | ⭐⭐ | DSP/FPGA |

## 6.8 本章小结

| 改进方向 | 核心思想 | 解决什么问题 | 嵌入式落地难度 |
|---------|---------|-------------|--------------|
| **模糊 PID** | 经验规则查表 | 非线性、难建模 | 低（查表法） |
| **增益调度** | 多组参数切换 | 时变、多工况 | 极低 |
| **MRAC** | 在线自适应 | 参数未知或漂移 | 中 |
| **NN/RL-PID** | 数据驱动 | 强非线性、复杂系统 | 中高（推理可部署） |
| **分数阶 PID** | 分数阶微积分 | 记忆效应、精细调相 | 高 |
| **解耦控制** | 结构解耦 | MIMO 耦合 | 中（简化前馈） |

> **工程师的清醒**：改进算法是"锦上添花"，不是"雪中送炭"。如果经典 PID 调不好，先检查传感器、执行器、采样周期、数值稳定性——**硬件和基础结构的问题，用再高级的算法也救不了**。
